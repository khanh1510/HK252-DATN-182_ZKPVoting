## Contracts

### `contracts/merkle/PoseidonT3_Vendor.sol`

Library `Bn254Poseidon2` — implements Poseidon T=3 (2 inputs, 1 output) on the BN254 curve.

Why this is needed on-chain:
- The Merkle Tree stores voter commitments on-chain using Poseidon to hash nodes.
- Must use the **same hash function** as the off-chain circuit so the on-chain root matches the root inside the proof.
- Keccak-256 (native EVM) cannot be used because the circuit uses Poseidon.

Implemented in **inline assembly** for gas efficiency. Single function:

```solidity
library Bn254Poseidon2 {
    function hash(uint256[2] memory inputs) internal pure returns (uint256)
}
```

Must be **deployed separately** and **linked** when compiling `IncrementalBinaryTree`. The deploy script handles this automatically.

### `contracts/merkle/IncrementalBinaryTree.sol`

Library `IncrementalBinaryTree` — manages an incremental binary Merkle Tree on-chain.

Why incremental: voters register over time, each time inserting a new leaf. The tree must update efficiently without storing all 2^depth nodes.

`IncrementalTreeData` struct:

```solidity
struct IncrementalTreeData {
    uint8   depth;
    uint256 root;
    uint256 numberOfLeaves;
    uint256[MAX_DEPTH] zeroes;       // precomputed zero hash at each level
    uint256[MAX_DEPTH] lastSubtrees; // last left node at each level
}
```

Two main functions:

```solidity
function init(IncrementalTreeData storage self, uint8 depth, uint256 zero) internal
function insert(IncrementalTreeData storage self, uint256 leaf) internal
```

`insert` uses `lastSubtrees[]` to track the last left node at each level. On new leaf insertion it hashes upward, updating only O(log N) nodes to reach the new root.

**Compatibility:** matches `@zk-kit/incremental-merkle-tree` in JavaScript exactly — the root computed off-chain (in the browser) always equals the on-chain root.

### `contracts/interfaces/IGroth16Verifier.sol`

Interface used by `VotingPool` to call the verifier:

```solidity
interface IGroth16Verifier {
    function verifyProof(
        uint256[2] calldata _pA,
        uint256[2][2] calldata _pB,
        uint256[2] calldata _pC,
        uint256[7] calldata _pubSignals
    ) external view returns (bool);
}
```

`VotingPool` holds a reference typed as `IGroth16Verifier` — independent of the concrete implementation. Allows swapping between real and mock verifier without modifying the contract.

### `contracts/Groth16Verifier.sol`

**Auto-generated** by:

```bash
snarkjs zkey export solidityverifier build/vote_final.zkey Groth16Verifier.sol
```

**Do not edit this file manually** — any changes will be overwritten on the next export.

Contents:
- Hardcoded verification key constants: vk_alpha ∈ G1, vk_beta/vk_gamma/vk_delta ∈ G2.
- 8 IC points ∈ G1 (one constant + one per public signal, for 7 public signals).
- `verifyProof(pA, pB, pC, pubSignals)` — performs the pairing check via EVM precompiles `0x06`, `0x07`, `0x08`.

**Note on pB coordinate reversal:** snarkjs outputs `pi_b = [[x0,y0],[x1,y1]]`, but EVM precompile 0x08 expects `[[y0,x0],[y1,x1]]`. `proof.ts` in the Dapp handles this swap before calling `castVote`.

### `contracts/mocks/MockGroth16Verifier.sol`

Stub for tests, implements `IGroth16Verifier`:

```solidity
contract MockGroth16Verifier is IGroth16Verifier {
    bool public immutable alwaysValid;
    constructor(bool _alwaysValid) { alwaysValid = _alwaysValid; }
    function verifyProof(...) external view returns (bool) { return alwaysValid; }
}
```

- `alwaysValid = true` — every proof passes → test contract logic without generating real proofs (saves 8–15s per test case).
- `alwaysValid = false` — every proof fails → test `InvalidProof` error paths.

### `contracts/VotingPool.sol`

Core contract managing the full lifecycle of one poll. Inherits `Ownable` and `EIP712` from OpenZeppelin.

#### State Machine

```
Phase.Registration
    │  startVoting()   [onlyOwner]
    ▼
Phase.Voting
    │  startReveal()   [onlyOwner]
    ▼
Phase.Reveal
    │  endPoll()       [onlyOwner]
    ▼
Phase.Ended
```

`currentPhaseActual()` returns the effective phase: if a deadline has passed but the owner hasn't called the transition yet, this view auto-advances the phase — used by the dApp to show the correct state.

#### Voting Types

Set at construction via `totalVotes` and `maxPerCandidate` (both `uint8`, range 1–255):

| totalVotes | maxPerCandidate | Mode |
|---|---|---|
| 1 | 1 | Single choice |
| K | 1 | Multiple choice |
| N | N | Cumulative |
| N | C | Cumulative with cap |

#### Weighted Voting (`isWeighted = true`)

Admin can call `setWeight(nullifierHash, weight)` after a voter registers to assign a multiplier applied to all their vote counts at reveal time. The voter's `nullifierHash` is emitted in the `VoterRegistered` event during `register()`.

#### `register(commitment, nullifierHash, adminSig, sigDeadline)`

Registers a voter by inserting their commitment into the Merkle Tree.

**OPEN mode:** checks phase, deadline, and `hasRegistered[msg.sender]`.

**ADMIN_APPROVED mode:** additionally calls `_verifyAdminApproval()`:
1. Checks `block.timestamp <= sigDeadline`.
2. Computes EIP-712 digest: `hash(VOTER_APPROVAL_TYPEHASH, voter, pool, deadline)`.
3. Checks digest not yet used (`usedApproval[digest]`), marks it used.
4. Recovers signer from signature → must equal `owner()`.

The digest binds to `address(this)` — a signature for pool A cannot be replayed on pool B.

#### `castVote(pA, pB, pC, pubSignals)`

Commits a hidden ballot in the Voting phase. Checks in order:

1. `currentPhase == Voting` and `block.timestamp <= votingDeadline`.
2. `pubSignals[0]` (merkleRoot) == `_tree.root`.
3. `pubSignals[3]` (numCandidates) == `realCandidates + 1`.
4. `pubSignals[4]` (totalVotes), `pubSignals[5]` (maxPerCandidate), `pubSignals[6]` (allowAbstain) match immutable config.
5. `nullifierUsed[nf] == false`.
6. `verifier.verifyProof(pA, pB, pC, pubSignals) == true`.

#### `revealVote(nullifierHash, votes[8], expectedCommit)`

Reveals a ballot in the Reveal phase. Checks:

1. Phase, deadline, not yet revealed, nullifier was used (cast).
2. `voteCommitmentOf[nullifierHash] == expectedCommit`.
3. Per-slot sanity: each slot ≤ `maxPerCandidate`, abstain exclusive, total ≤ `totalVotes`.

`expectedCommit = Poseidon(votes[0..7], blinding)` computed off-chain by the voter. If they claim a different allocation than what they committed to, the check fails. Poseidon collision-resistance makes it infeasible to find a different `votes'` that produces the same commitment.

Vote counts are accumulated as `votes[i] * weight`.

#### Custom Errors

| Error | When |
|-------|------|
| `InvalidPhase` | Action called in wrong phase |
| `DeadlineExpired` | Phase deadline has passed |
| `AlreadyRegistered` | Address registered twice |
| `InvalidApproval` | EIP-712 signature has wrong signer |
| `ApprovalExpired` | Signature past deadline |
| `ApprovalAlreadyUsed` | Signature replayed |
| `InvalidProof` | Groth16 verification returned false |
| `MerkleRootMismatch` | Root in proof ≠ on-chain root |
| `NumCandidatesMismatch` | numCandidates in proof ≠ poll config |
| `TotalVotesMismatch` | totalVotes in proof ≠ poll config |
| `MaxPerCandidateMismatch` | maxPerCandidate in proof ≠ poll config |
| `AllowAbstainMismatch` | allowAbstain in proof ≠ poll config |
| `DoubleVote` | Nullifier already used |
| `AlreadyRevealed` | Reveal called twice |
| `CommitmentMismatch` | expectedCommit ≠ stored commitment |
| `InvalidVotes` | Vote allocation violates rules |
| `ResultsNotAvailable` | `getResults()` before Ended phase |
| `InvalidPhaseTransition` | Phase transitions out of order |
| `InvalidConfig` | Invalid constructor args |
| `NotWeightedPoll` | `setWeight()` called on non-weighted poll |

### `contracts/PollFactory.sol`

Factory contract — deploys new VotingPool instances on demand and stores a registry.

```solidity
function createPoll(
    VotingPool.EligibilityMode mode,
    string calldata proposal,
    string[] calldata candidateNames,
    uint256 registrationDeadline,
    uint256 votingDeadline,
    uint256 revealDeadline,
    uint8   totalVotes,
    uint8   maxPerCandidate,
    bool    allowAbstain,
    bool    isWeighted
) external returns (uint256 pollId, address pool)
```

`msg.sender` becomes the `owner` of the new VotingPool. Emits `PollCreated(pollId, pool, admin, mode, proposal)`.

One `Groth16Verifier` is shared across all pools — deployed once, saving ~1.5M gas vs. each pool deploying its own verifier.

## Deploy

```bash
npx hardhat run scripts/deploy.js --network sepolia
```

Deploy order in `scripts/deploy.js`:
1. Deploy `Bn254Poseidon2` library.
2. Deploy `Groth16Verifier` or `MockGroth16Verifier` (based on `USE_REAL_GROTH16_VERIFIER`).
3. Deploy `PollFactory` with library linking + verifier address.
4. If `SAMPLE_POLL=true`: call `factory.createPoll()`.
5. Save all addresses to `deployments/<network>.json`.

## Tests

```bash
npx hardhat test
REPORT_GAS=true npx hardhat test
```

`test/VotingPool.test.js` uses `MockGroth16Verifier(true)`. Coverage:

- **OPEN mode:** register, no double register, deadline enforcement, castVote, double-vote prevention.
- **ADMIN_APPROVED mode:** valid EIP-712 coupon, missing signature rejected, replay attack, expired coupon.
- **Phase machine:** owner-only transitions, correct ordering, `getResults` gated by Ended.
- **revealVote:** commitment match, mismatch, double reveal, invalid vote allocation.
- **Weighted voting:** `setWeight`, weight applied at reveal, default weight = 1.
